## Введение к разделу 4 (1 абзац в Word)

В разделе приведены схемы алгоритмов и структурная схема программного комплекса **АОИ-Web**. Блок-схемы алгоритмов выполнены по **ГОСТ 19.701-90** (процесс — прямоугольник, решение — ромб, начало/конец — овал). Структурная схема программы отражает состав подсистем и потоки данных между клиентом, сервером и хранилищем.

## Рисунок 4.1 — Структурная схема программы

### Текст под рисунком

На рисунке приведена структурная схема программного комплекса **АОИ-Web** на уровне основных компонентов. **Веб-клиент** (HTML/CSS/JavaScript) взаимодействует с **сервером приложений FastAPI** через **REST API** (HTTPS, JSON). На сервере выделены подсистемы: аутентификация и роли, инспекции, устройства и видеопоток, датасеты, настройки, эталоны **Golden Board**, статистика и отчёты. Уровень хранения включает **СУБД** (SQLite или PostgreSQL, доступ через **SQLAlchemy ORM**) и **файловое хранилище** `storage/` для изображений и артефактов. Стрелками показаны основные потоки запросов и обмен данными.

## Рисунок 4.2 — Алгоритм создания инспекции

### Текст под рисунком

Схема описывает обработку запроса на инспекцию одного кадра. После проверки аутентификации и параметров изображение проходит предобработку (OpenCV). При указании профиля Golden Board выполняется попытка ECC-выравнивания; при неуспехе используется исходный кадр. Модель YOLO формирует список детекций. Для компонентных классов выполняется проверка наклона (`placement_tilt`). Если выравнивание по эталону было применено и в профиле заданы зоны `regions`, выполняется сверка детекций с разметкой эталона с учётом допуска по положению; при расхождении добавляются дефекты «компонент не найден» или «не тот компонент». Результат сохраняется в СУБД и файловом хранилище, клиенту возвращается JSON-ответ.

## Рисунок 4.3 — Алгоритм выравнивания Golden Board (ECC)

### Текст под рисунком

Схема описывает опциональный этап совмещения текущего кадра с опорным снимком эталона. При отсутствии идентификатора профиля этап пропускается. При указании профиля выполняется загрузка данных из СУБД и файлового хранилища; при отсутствии профиля или файла эталона формируется ответ об ошибке. Алгоритм ECC оценивает качество выравнивания по средней абсолютной ошибке (MAE). При недостаточном улучшении обработка не прерывается: используется исходный кадр, в метаданных фиксируется `applied=false`. При успехе — выровненный кадр для последующей детекции и сверки зон эталона.

### Промпт для рисунка (копировать целиком)

```
Сгенерируй ИЗОБРАЖЕНИЕ: блок-схема по ГОСТ 19.701 (овал, прямоугольник, ромб, стрелки,
подписи «да/нет»). Монохром или 2 цвета. Фон белый, альбомная A4, вертикальный поток,
минимум пересечений. Русский текст, крупный шрифт.

Типографика: текст не касается рамок фигур; padding 10–14 % высоты; в ромбе переносы строк;
без микрошрифта. Высота фигуры минимум в 1,4 раза больше текста.

Содержание (важно: «мягкий отказ» — пайплайн продолжается без аварии):

1) Овал: «Начало: этап Golden Board (ECC)»

2) Ромб: «Указан golden_board_profile_id?»
   нет → Прямоугольник: «Пропуск этапа» → Прямоугольник: «Передать исходный кадр дальше» → Овал «Конец этапа»
   да → Прямоугольник: «Загрузить профиль из СУБД и опорный снимок из storage/»

3) Ромб: «Профиль и файл эталона найдены?»
   нет → Прямоугольник: «Ответ об ошибке (HTTP 404)» → Овал «Конец (ошибка)»
   да → Прямоугольник: «ECC-выравнивание текущего кадра к эталону»

4) Ромб: «MAE после выравнивания лучше порога?»
   нет → Прямоугольник: «applied = false, исходный кадр»
   да → Прямоугольник: «applied = true, выровненный кадр»

5) Прямоугольник: «Передать кадр на детекцию YOLO»

6) Овал: «Конец этапа»

Запреты: без «4хх»; без логотипов; без кода на фоне.

Подпись под рисунком:
«Рисунок 4.3 — Алгоритм выравнивания по эталону Golden Board (ECC)»
```

## Рисунок 4.4 (опционально) — Подготовка разметки эталона

Используй, если нужен отдельный рисунок про вкладку «Эталоны». Не обязателен при лимите страниц.

### Текст под рисунком

Схема описывает подготовку профиля Golden Board администратором. Создаётся профиль эталона, загружается опорный снимок. При включённой авторазметке YOLO формирует зоны `regions` с классами модели; дефекты поверхности платы отфильтровываются. Оператор корректирует зоны и классы в интерфейсе и сохраняет разметку в `payload_json` профиля.
